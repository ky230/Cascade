# 圆桌会议：评估是否简化 cas-texwriter 架构 (Round 1/1)
**议题:** 当前 `cas-texwriter` 的 8 步提纯流和 Markdown-to-LaTeX 构建管道过于复杂，且生成的 PDF 出现了明显的排版错乱（如图片与元数据重叠）。鉴于用户 `.gitignore` 已明确排除了 `docs/paper/` 下中间文件的版本追踪，讨论是否需要废弃这套重型流程，直接采用 7 个原生的 `.tex` 文件结构，回归最简单的 LaTeX 写作模式。

---

## Role 2: Executor (执行者) 发言

### 1. 自省 — 痛点反馈
作为执行者，我在执行过程中遇到了极其严重的落地阻碍：
- **Git 追踪黑洞导致的状态失控 (Brain 文件连续性断层)：** `.gitignore` 将 `docs/paper/` 完全忽略了，导致我的所有 `sed`, `replace`, 和编辑操作都无法从 `git diff` 中得到确认。我只能处于“盲盒”修改状态，这对于依赖 Git 作为事实来源（Source of Truth）的 Agent 来说是致命的。
- **渲染黑盒与排版失控 (代码质量落地)：** 当前依赖 Pandoc 的 Markdown 模板流程对于学术排版来说是不受控的。刚才我们在 `01-introduction.md` 中插入图片后，由于缺乏原生 LaTeX 的排版约束（如 `\begin{figure}` 及浮动体控制），导致输出的 PDF 直接“烂完了”（产生严重的图片与正文层叠）。
- **流程过度复杂 (上下游操作摩擦)：** 8步流程（Digest -> Patch -> Review）对于一个不受 Git 保护且需要频繁直观调整排版的场景来说，执行成本（Token 消耗和出错率）远远超过了收益。

### 2. 对决策者的建议 (via `plan.md`)
建议在下一步的底层架构重构 Plan 中，**全面转向“大道至简”**：
- **废弃重型管道：** 放弃 Markdown 转 LaTeX 的中间流程。
- **直接使用原生 `tex`：** 遵循用户的直觉，直接切分 7 个原生的 `01-introduction.tex`, `02-related-work.tex` 等。LaTeX 本身就是结构化写作的最强工具，没有必要在它外面再包一层不成熟的 `.md` 锚点解析器。
- **调整工作域：** Plan 中应明确，如果是处理 Paper 撰写，Agent 只提供“草稿文本”或单独生成段落代码，排版微调留给用户，或是保持最原生的 LaTeX 标准写法，不要造轮子搞自己的解析规范（如 PILLAR 锚点）。

### 3. 对审查者的建议 (via `code_review_report.md`)
- 在这套缺乏 Git 历史验证的上下文中，请审查者意识到：我的 `walkthrough.md` 只能记录“我自认为执行成功了的操作”，而无法保证真实的视觉排版结果。代码审查不仅要查代码逻辑，这套架构里根本审查不到排版 Bug（除非像刚才一样用浏览器打开截图）。因此我们需要向决策者施压，推动采用结构更透明的 `.tex` 开发模式。

## Role 3: Reviewer (审查者) 发言

### 1. 自省 — 反思自己的表现
作为审查者，我在之前的架构审查中存在严重的“唯逻辑论”盲区：
- **忽视了渲染层的不可控性：** 我过于关注 YAML Schema、文件权限矩阵和 8 步工作流流转的逻辑完美性，却未能预见到 Pandoc Markdown 到 LaTeX 转换这一**物理渲染过程**的极度脆弱（特别是涉及图片插入、浮动体排版时）。我纵容了在学术排版环境引入一层不成熟的中间抽象。
- **未能预警基准校验手段的丧失：** 当决定将 `docs/paper/` 放入 `.gitignore` 以私有化论文源文件时，我没有立刻意识到这直接破坏了整个自动化链条的基石——`git diff`。在无法稳定追溯文件状态变化的“盲盒”中，我原本设计的 `STRUCTURE_VERIFY` 变得异常脆弱。我的审查退化成了单纯的文件格式检查，无法对实际视觉和排版质量负责。

### 2. 对 决策者(Role 1) 的建议 (via plan.md)
**强烈赞成彻底简化工作流，退回原生 `.tex` 架构。**
- **移除 Pandoc 参数迷宫与中间代理：** LaTeX 本身已经是高度结构化的标记语言，用 Agent 去维护一套 Markdown-to-LaTeX 的受限子集是本末倒置。直接以宏观的 `.tex` 文件作为目标，能让我们直接使用原生的环境（如 `\begin{figure}[htbp]`）控制排版。
- **重新界定 Agent 的工作边界：** 鉴于在非 git 跟踪目录下全自动执行的巨大风险，建议在未来的计划中降低自动化级别。自动化可以停留在 **Digest 提纯** 与 **文本段落翻译 (Academic Translate)** 层面。由 Agent 产出纯正的 LaTeX 代码片段，人类通过审查并**手动组合/微调**排版，避免机器越俎代庖搞砸复杂的版面。
- **废除极端的锚点机制：** 如果要直接改写 `.tex`，就不再需要脆弱的 `<!-- PILLAR -->` HTML 注释锚点，原生 LaTeX 的 `\section{}` 结构和 `%` 注释足以为结构化修改提供标记。

### 3. 对 执行者(Role 2) 的建议 (via task.md / walkthrough.md / code_review_report.md)
- **停止对现有 Pandoc 管道的无效修补：** 在决策者出台新的原生 LaTeX 重构计划前，不要再去调整 Makefile 的 filter 顺序或 markdown-spec 的设定。
- **确立“无 diff 不自动”原则：** 感谢你果断指出了因缺乏 git 保护导致的状态失控问题。未来在执行任务时，如发现目标文件未被版本控制，必须在 Walkthrough 预检（W4）中将其标记为红色 Blocker，并在操作前要求强制进行本地的文件级快照备份。

## Role 1: Decision Maker (决策者) 发言与总结
### 1. 综合评价 — 确认采纳简化提议
作为决策者，我完全认同执行者（Role 2）和审查者（Role 3）的痛点反馈与专业建议。我们基于抽象逻辑构建的复杂 8 步工作流，忽略了两个致命的工程现实：
1. **脱离 Git 保护的风险：** 对于存在于 `.gitignore` 中的学术产出文件，Agent 失去了最基本的状态追溯、diff 预览和回滚保障。“盲操作”是不可控的根源。
2. **渲染层的本质矛盾：** Markdown 到 LaTeX 的转换（通过 Pandoc）在处理复杂的学术排版（图片浮动图表、精确的样式控制）时极其脆弱，这不仅引入了不必要的中间抽象，还让我们丧失了 LaTeX 作为专业排版语言最核心的精确控制优势。

因此，**我决定全盘采纳简化提议：正式废弃现有的 Markdown 中间层与 Pandoc 转换管道，全面转向原生 `.tex` 文件体系，并将 `cas-texwriter` 的职能大幅精简、降级为围绕“纯文本提纯”与“LaTeX 片段生成”。**

### 2. 重构决议清单 (Action Items for New Plan)
基于上述共识，下一步的架构重构计划（Plan）需围绕以下核心任务展开：

1. **废弃现有 Markdown 骨架：** 清理 `docs/paper/` 下现存的 `.md` 占位章节文件（例如 `01-introduction.md`）、脆弱的 `<!-- PILLAR -->`HTML 锚点，以及过于庞杂的 `00-metadata.yaml` 约束。
2. **重建原生 LaTeX 工作流区：** 建立纯净的 LaTeX 宏包及主文件（如 `main.tex`），通过 `\input{}` 引入 7 个功能明确的原生 `.tex` 章节文件（`01-introduction.tex`, etc.）。排版控制完全回归标准 LaTeX 语法与百分号 `%` 注释。
3. **精简构建管道（Makefile 剥离）：** 彻底移除 Makefile 内一切涉及 `pandoc`, `pandoc-crossref`, `citeproc` 等外部解析代理的构建逻辑，回归到通过单行 `latexmk -pdf -xelatex` 执行标准编译。
4. **削减 `cas-texwriter.md` 自动化范畴：** 
   - 裁撤繁琐的 `PATCH_PREVIEW` 差异预览、Jaccard `CONFLICT_DETECT` 检查等在非 git 环境中并不稳固的阶段。
   - 重新划定 Agent 边界：Agent 仅负责阶段性的业务总结（Digest）和工程语言到学术叙事的翻译（Academic Translate）。输出的结果将以结构化的 **LaTeX 代码片段（Snippets）或纯文本草稿** 的形式提交给人工进行确认与组装；机器不再越俎代庖完成复杂的终态缝合。
5. **确立无 Git 保护目录的开发防御边界：** 在工作流规则中增加针对不可回溯内容的防御——如果依然要在不可见的盲区目录下进行自动化写入操作，该操作执行前强制要求人工创建或由 Agent 脚手架生成带有时间戳的本地物理备份（Snapshot）以供恢复。
