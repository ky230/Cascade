# 设计文档：AI 生成期间的并发输入处理机制

**目标**：解决在模型生成（Streaming）时因并发输入造成的堵塞报错，提供顺畅的多轮连续输入体验。

通过交叉对比 `Gemini CLI` 和 `Claude Code` 的源码，发现目前业界对“生成期后续输入”主要有三种截然不同的设计哲学。以下是三种方案的权衡分析：

---

## 方案 A：Gemini CLI 模式（合并批处理 Batching）
**机制**：
- 用户在生成期间如果多次按 Enter，所有文本依次存入 `messageQueue`。
- 当 AI 当前这轮生成一结束（转为 Idle），系统会将队列中的所有文本用 `\n\n` 拼接成一个超级长文本（Combined Prompt）。
- 清空队列，并将这个长文本作为**一次独立的新提问**发给大模型。

* **优点**：节省 API Request 成本，减少生成碎片。能让模型一次性拥有完整的“追加上下文”。
* **缺点**：不适合混合系统指令（例如队列里有 `/model` 或者 `/clear` 时，把它们和普通文本拼合会导致命令解析器懵逼或报错）。

## 方案 B：Claude Code 模式（单线顺序投递 Sequential）
**机制**：
- 用户按 Enter 后，命令存入待办队列。
- AI 当前生成结束时，从队列队首取出**仅 1 条**指令，启动新一轮执行链路。结束后再拿下一条。

* **优点**：最严谨、最纯正的按序执行系统。对 Slash Commands（ `/` 命令）最安全，就像用户排队一次敲入一行一样。
* **缺点**：如果有三条很短的补充，会被强行拆成三次完整的、带延迟的 LLM 往返调用。

## 方案 C：强力打断模式（Interrupt & Override）
**机制**：
- （非典型的默认队列，但很多现代 IDE 喜欢）用户一按 Enter 发送新指令，立刻 `cancel()` 强杀掐断 AI 当前还在生成的胡言乱语。
- 立即用新发出的 prompt 开启新一轮的生成。

* **优点**：最适合纠错（“Vibe Coding”时发现 AI 方向错了，直接发一句“不对，停下，用另外的方法”，立竿见影）。
* **缺点**：实现复杂度高（需要处理完善的 Task Cancellation 闭环以防止状态变量脏页）。

---

## 我的结论与推荐

考虑到我们刚刚花了大精力将 Cascade 打造成了一个有 37 个 Slash Commands 的系统，**方案 A（无脑合并）可能会导致斜杠路由器的灾难**（如果用户排队了两个 Slash 混合普通文本，就会解析失败）。

🏆 **最优推荐：方案 B（顺序执行）搭配手动打断（通过组合键或独立信号）**
- 将用户敲入的请求用真实的 `Queue` 排队，逐条发送以确保 `/命令` 的精确解析。
- 而打断功能（如果是需求）则应该像 Claude Code 的 `* Burrowing... (ctrl+c to interrupt)` 那样，独立占用一个热键，而不是默认由 Enter 触发。
